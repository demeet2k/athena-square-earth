# GOVERNANCE AND PROMOTION

## 1. Purpose

The CPU must not turn compression into hallucination or expansion into chaos. Governance
exists to keep every wave replayable and to keep every final artifact honest about what
is known, inferred, or still unresolved.

## 2. Truth Classes

Every packet is marked:

- `OK`
- `NEAR`
- `AMBIG`
- `FAIL`

No packet may skip classification.

## 3. Promotion Rules

Promotion from `AMBIG` or `NEAR` to `OK` requires:

1. direct source support,
2. stable contraction lineage,
3. no outstanding Earth veto,
4. one replay hint that another pass could follow,
5. one explicit statement of what was discarded.

## 4. Conflict Law

When two packets disagree:

1. preserve both packets,
2. identify whether the conflict is:
   - lexical,
   - structural,
   - evidentiary,
   - interpretive,
3. route the conflict to the relevant archetype reviewer,
4. only collapse the conflict if a witness-bearing discriminator exists.

If no discriminator exists, the correct status is `AMBIG`.

## 5. Overreach Guardrail

The CPU may infer architecture from repeated patterns, but it must not present an
unwitnessed inference as direct evidence. Every final synthesis must separate:

- direct source claim,
- derived structural inference,
- open hypothesis.

## 6. Source Priority

Priority order for this workspace:

1. live Google Docs, when the gateway is available,
2. local extracted text mirrors,
3. canonical manuscript crystals and syntheses,
4. secondary summaries.

If the live Docs gateway is blocked, the failure reason should be recorded in the task
ledger before local fallback begins.

## 7. Release Rule

A CPU artifact is release-ready only when:

1. its final packet is `OK` or honestly marked `NEAR`,
2. all source paths are preserved,
3. the contraction path is inspectable,
4. unresolved risks are listed plainly.
