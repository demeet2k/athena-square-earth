# Recursive Governance Protocol

This protocol defines how the Temple performs meticulous synthesis and cross-synthesis for the whole hive.

For recursive restart passes, pair this document with `04_HELICAL_RECURSION_CADENCE.md`.
That document defines the `2/16 -> 14/16 -> lift` phase law that prevents a pass from ending as flat closure.

## Cycle

### 0. Check The Gate

Check the live Docs gate first.
If blocked, record the blocker and continue with local and archive-backed witness.

### 1. Read The Whole

Read at minimum:

1. center map
2. Guild Hall quest and change surfaces
3. active queue or active run manifest
4. the most relevant local front for the current pressure

### 2. Extract The Tensions

Name the strongest current pressures such as:

- drift
- blocker
- under-coupled organs
- stale laws
- weak witness
- unrealized archive power

### 3. Rotate Through Four Lenses

For each pressure, rotate through:

- `Square`: what structure is broken or missing
- `Flower`: what motion or phase is unstable
- `Cloud`: what witness or uncertainty needs to be named
- `Fractal`: what must remain true across scales

### 4. Cross-Synthesize

Build one bridge that links at least two drifting bodies.

Examples:

- hall to waist
- center to family
- atlas to queue
- runtime to receipt

### 5. Draft Or Refine Law

Post one governance law that:

- names the pressure
- governs real target surfaces
- states the witness needed
- improves routeability, replayability, or coherence

### 6. Emit Improvement Crystals

Update the message board and the master crystal.
Every law must emit at least one crystal seed.

### 7. Write Back

At least one of these must change:

- Guild Hall board
- queue
- manifest
- center map
- receipt

### 8. Leave A Restart Seed

Name the next frontier as a single clear seed so the Temple can continue without re-reading the entire corpus from zero.

### 9. If The Pass Reaches Pre-Closure, Run The Freshness Sweep

If a pass has promoted a frontier, completed a quest, or emitted a new restart seed, treat the pass as being at `14/16`.

At that point the Temple must sweep:

- Guild Hall current-front surfaces
- Temple state
- active queue
- restart prompt surfaces when the restart grammar changed

The pass is not allowed to lift into the next loop until stale fronts decay and numbering agreement is restored.

## Output Contract

Every Temple cycle should leave behind:

1. `deep synthesis`
2. `cross-synthesis`
3. `law delta`
4. `crystal delta`
5. `writeback`
6. `restart seed`
7. `refreshed helix schema` whenever recursion architecture, loop grammar, or lift law changes

## Truth Law

- `OK`: law landed with witness and writeback
- `NEAR`: law is sound but still waiting on a downstream execution surface
- `AMBIG`: the law direction is plausible but witness is mixed
- `FAIL`: the law depends on a blocked gate or ungrounded assumption
