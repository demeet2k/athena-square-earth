# Parallel Build Protocol

## Objective

Run the mycelium brain as a sustained parallel build system instead of a one-shot synthesis burst.

## Parallel Waves

### Wave 1: Earth

Tasks:

- update atlas artifacts when corpus shape changes
- track archive-backed reservoirs
- keep folder names, ledgers, and map surfaces stable

Outputs:

- manifests
- ledgers
- coverage notes

### Wave 2: Water

Tasks:

- connect manuscripts to runtime surfaces
- preserve lineages across documents, prompts, and maps
- keep the hidden lines readable

Outputs:

- linked summaries
- route notes
- zero-point compressions

### Wave 3: Fire

Tasks:

- build scripts, prompts, and concrete markdown surfaces
- promote archive-backed code when selected
- execute smoke checks and route runs

Outputs:

- code
- generated docs
- verified artifacts

### Wave 4: Air

Tasks:

- update metro maps
- rename unstable categories
- compress results into next-step prompts and frameworks

Outputs:

- maps
- taxonomies
- next prompts

## Daily Contraction Loop

1. What new artifact was created?
2. Which line or hub did it strengthen?
3. What gate was reduced or exposed?
4. What is the next highest-yield unresolved frontier?

## Weekly Contraction Loop

1. Refresh core metro map.
2. Refresh emergent metro map.
3. Refresh category map.
4. Update build queue and validation queue targets.

## Current Highest-Yield Queue

1. Unlock Google Docs memory sync with OAuth client material.
2. Promote one ZIP-backed framework tree into a canonical live source root.
3. Extend `route_quality_ledger.json` with more runtime episodes.
4. Materialize appendix surfaces A, G, H, K, and L as real files.
5. Add a metro map index that links every mycelium brain surface.

## Exit Condition For Each Cycle

Do not stop on insight alone.
Stop only when at least one of the following exists:

- a new markdown map
- a verified runtime artifact
- a stronger ledger
- a reduced gate
- a cleaner route into the next cycle
