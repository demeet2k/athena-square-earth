# Parallel Execution Loop

Use this as the per-run operating loop for any single agent working inside the shared nervous system.

## Loop

1. Read `06_active_queue.md`.
2. Read `registry/01_tandem_frontier_claims.md`.
3. Read `09_higher_dimensional_mapping.md` and `10_deeper_emergent_neural_swarm.md`.
4. Read `12_recursive_restart_engine.md`, `14_37_gate_recursive_escalation.md`, and `15_256x256_corpus_hyperplan.md`.
5. Attempt the live Docs search if the front is a new manuscript section or chapter.
6. If blocked, record the exact query and error, then continue with local mirrors.
7. Choose the smallest frontier that yields the strongest shared capability gain.
8. Express that frontier as an `NSCoord` and place it in the swarm:
   - family
   - rail
   - face
   - scale
   - hub
   - regime
   - packet class
   - truth class
9. Produce one routed artifact, not just analysis.
10. Write a receipt, queue update, or ledger entry before stopping.
11. Emit the next restart seed so the loop re-enters at step 1.

## Front Selection Rule

Prefer fronts in this order:

1. gate unlocks
2. family tensor strengthening
3. archive-to-live promotion
4. manuscript family routing
5. runtime packet or ledger writeback
6. contraction of parallel shard output
7. map refinement

## Gate 37 Rule

If perceived completion is high and external satisfaction drops sharply:

1. classify the event as a Gate 37 escalation
2. read `14_37_gate_recursive_escalation.md`
3. identify the weakest gate band still below target strength
4. land one artifact that upgrades that band
5. restart from the beginning instead of polishing the wrong layer

## Stop Rule

Do not stop on raw insight alone.
Stop only after one of the following lands:

- a new markdown artifact
- a stronger receipt
- a stronger claim ledger
- a stronger queue
- a stronger runtime ledger entry
- a stronger packet, manifest, or family surface

## Drift Rule

If you notice yourself expanding sideways without improving shared reuse, contract immediately and land a smaller artifact.
