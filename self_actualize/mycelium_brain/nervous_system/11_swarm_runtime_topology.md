# Swarm Runtime Topology

## Purpose

The deeper neural swarm needs a real filesystem anatomy, not only descriptive markdown.
This topology is the live runtime body of the nervous system.

## Runtime Folders

### `families/`

Family-level tensor and ownership surfaces.

### `ganglia/`

Durable family-local cluster notes.

### `neurons/`

Reusable interconnecter neuron packets.

### `packets/`

Runtime packets, pod packets, and synthesis packets.

### `rails/`

Long-range control lane surfaces.

### `hubs/`

Metro hub index and local hub notes.

### `manifests/`

Active-run, next-prompt, and restart-token surfaces.

### `swarms/`

One file per active swarm specification.

### `waves/`

One file per synchronized execution pass.

### `ledgers/`

Witness, ambiguity, conflict, route, and promotion ledgers.

### `sessions/`

Session handoff packets for restart continuity.

### `receipts/`

Major operation receipts local to this branch.

### `DEEPER CRYSTALIZATION\ACTIVE_NERVOUS_SYSTEM\07_FULL_PROJECT_INTEGRATION_256\06_REALTIME_BOARD\`

Canonical coordination surface for live claims, agent inboxes, active threads, and workspace change observation.

## Minimal Swarm Run

Every serious swarm run should leave:

1. one `SWARM_*.md`
2. one `WAVE_*.md`
3. one packet or pod packet
4. one synthesis, manifest, or session token
5. one receipt or ledger writeback
6. one queue or index update

## Packet Schema

Every serious runtime packet should expose:

- front
- family
- lineage or address
- scale
- rail
- hub
- packet class
- truth
- output target
- witness target
- contraction target
- next seed

## Runtime Law

The runtime branch is healthy only if:

- family surfaces exist
- ganglia and neurons agree on ownership
- rails name long-range routing
- manifests carry restart seeds
- packets can be contracted into cortex
