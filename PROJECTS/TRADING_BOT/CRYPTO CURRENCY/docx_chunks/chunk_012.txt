def order_hubs_deterministically(hubs_set: FrozenSet[Hub], pa: ParsedAddr, intent: Intent, tau0: Truth, rp: RouterProfile) -> Tuple[Hub, ...]:
    # Build templated hubs (string tokens allow placeholder names in canonical_template)
    computed: Dict[str, Optional[Hub]] = {
        "AppA": "AppA",
        "ArcHub": rp.arc_hub.get(pa.station.arc) if (pa.is_chapter and pa.station is not None) else None,
        "LensBase": rp.lens_base.get(pa.lens),
        "FacetBase": rp.facet_base.get(pa.facet),
        "Overlay": truth_overlay(intent, tau0, rp),
        "AppI": "AppI",
        "AppM": "AppM",
    }
    seq: List[Hub] = []
    seen = set()
    for token in rp.canonical_template:
        h = computed.get(token, token)  # allow direct hub names or tokens
        if h is None:
            continue
        if h in hubs_set and h not in seen:
            seen.add(h)
            seq.append(h)
    return tuple(seq)

def compute_obligations(intent: Intent, tau0: Truth) -> Tuple[str, ...]:
    obls = ["Obl.ParseCanon(AppA)", "Obl.CorridorType(AppI)", "Obl.ReplayClose(AppM)"]
    if tau0 == "NEAR":
        obls.append("Obl.ResidualLedger(AppJ)")
    elif tau0 == "AMBIG":
        obls.append("Obl.EvidencePlan(AppL)")
    elif tau0 in ("FAIL", "CONFLICT"):
        obls.append("Obl.ConflictPacket(AppK)")
    # intent-specific obligations can be added deterministically by a pinned table
    return tuple(obls)

def route(addr: str, intent: Intent, tau0: Truth, rp: RouterProfile, pins: Pins) -> RoutePlan:
    pa = parse_addr_strict(addr)

    hubs: List[Hub] = []
    # Base hubs from lens/facet
    hubs.append(rp.lens_base[pa.lens])
    hubs.append(rp.facet_base[pa.facet])

    # Arc hub for chapters
    if pa.is_chapter and pa.station is not None:
        hubs.append(rp.arc_hub[pa.station.arc])

    # Enforce mandatory Σ
    hubs.extend(list(rp.mandatory_signature))

    # Truth overlay
    overlay_hub = truth_overlay(intent, tau0, rp)
    if overlay_hub is not None:
        hubs.append(overlay_hub)

    # Cap enforcement
    hubs2, droplog, tau1 = enforce_cap(hubs, rp, tau0)

    # Ordering
    hubs_seq = order_hubs_deterministically(frozenset(hubs2), pa, intent, tau1, rp)

    # Obligations
    obligations = compute_obligations(intent, tau1)

    return RoutePlan(intent=intent, target=addr, scope=pa.scope, station_header=pa.station,
                     hubs_seq=hubs_seq, tau0=tau1, pins=pins, obligations=obligations, droplog=droplog)

The reference implementation is intentionally structured so that every degree of freedom (tables, ordering, drop priority, overlay mapping) is pinned inside the router profile, ensuring router purity and replayability. The hub cap is enforced as a hard contract; drops are logged; and cap‑enforced unwitnessed drops deterministically degrade intended truth to AMBIG (or FAIL if the corridor forbids AMBIG).

❀ Flower — Canonical Ride Selection as Constraint Projection with Lexicographic Minimality
11.F1 — The constraint system
Let the router constraints be:
(C_{\Sigma}): include (\Sigma),
(C_{\mathrm{cap}}): (\le 6) unique hubs pre-target,
(C_{\mathrm{rail}}): include mandatory overlay hub induced by ((\mathsf{Intent},\tau_0)),
(C_{\mathrm{base}}): include base hubs induced by (\mathrm{LensBase},\mathrm{FacetBase},\mathrm{ArcHub}) (as applicable),
(C_{\mathrm{order}}): impose canonical ordering.
A candidate hub ride is a finite sequence (\mathbf{h}\in\mathcal{H}^{\le 6}). The router contract is equivalently the computation of a canonical feasible point of this constraint system under pinned tie-break rules.

11.F2 — Alternating projection interpretation
Define projectors on a “hub multiset space”:
(P_{\Sigma}): add (\Sigma),
(P_{\mathrm{rail}}): add the overlay hub induced by ((\mathsf{Intent},\tau_0)),
(P_{\mathrm{base}}): add base hubs induced by the parsed address,
(P_{\mathrm{cap}}): apply deterministic cap enforcement with DropLog,
(P_{\mathrm{order}}): map multiset to ordered sequence via canonical ordering template and compression.
Then the router is the finite composition:
[
\mathbf{h}\star = P{\mathrm{order}}\circ P_{\mathrm{cap}}\circ P_{\mathrm{rail}}\circ P_{\Sigma}\circ P_{\mathrm{base}}(\varnothing),
]
with the understanding that each projector is deterministic under (\pi_R). The “projection” language here is not analytic; it is the formal statement that routing is constraint satisfaction via deterministic normalization.

11.F3 — Lexicographic minimality and deterministic tie-break
Whenever multiple feasible rides exist (e.g., due to optional intent hubs or alternative ArcHub refinements), the router selects the unique ride minimizing a pinned lexicographic objective:
[
\min_{\mathbf{h}\in\mathcal{F}}\ \Bigl(\mathrm{CapViol}(\mathbf{h}),\ \mathrm{DropCount}(\mathbf{h}),\ \mathrm{DropPriorityScore}(\mathbf{h}),\ \mathrm{OrderKey}(\mathbf{h})\Bigr),
]
where (\mathcal{F}) is the feasible set given the pinned base hubs and mandatory signature/rail constraints. This objective must be pinned as part of (\pi_R) (or equivalently realized by pinned deterministic drop+order functions). Any unpinned tie-break introduces nondeterminism and violates the router purity law.

11.F4 — Ride equivalence and witnessed commutation
If an optimization proposes a different hub visit ordering, it is legal only if commutation is witnessed. In this chapter’s contract, this means: the engine may reorder only when it possesses a replayable equivalence witness that the reordered ride yields identical outputs and identical truth typing under pinned profiles. Otherwise the canonical ordering stands. This enforces the rule that reordering commutative hub visits is rare and must be witnessed; it cannot be assumed.

12.1 ReplayCapsule and the replay harness contract
12.1.1 ReplayCapsule (\mathsf{RC}): the minimal import‑verifiable verifier packet
A ReplayCapsule is the smallest self‑contained unit enabling an independent verifier to re‑run a claim check deterministically. Define:
[
\boxed{
\mathsf{RC}:=(\mathrm{RCID},\mathrm{TargetAddr},\mathrm{Scope},\mathrm{Pins},\mathrm{EnvPins},\mathrm{Seeds},\mathrm{Inputs},\mathrm{Steps},\mathrm{ExpectedDigests},\mathrm{OutputsSpec},\mathrm{LogsSpec},\mathrm{SchemaVer})
}
]
where:
(\mathrm{TargetAddr}) is the (\mathrm{GlobalAddr}) node/edge being verified,
(\mathrm{Pins}) includes all profile digests required to interpret the claim (router/corridor/grammar/canonbytes/kernel/phase/bridge/control/container schema),
(\mathrm{EnvPins}) pins runtime/platform/dependency behavior sufficient for deterministic replay,
(\mathrm{Seeds}) pins any randomness (empty if randomness is forbidden for the capsule class),
(\mathrm{Inputs}) are embedded canonical bytes or digest‑referenced canonical bytes,
(\mathrm{Steps}) is a deterministic execution recipe,
(\mathrm{ExpectedDigests}) are intermediate and final digests binding the claimed outcome,
(\mathrm{OutputsSpec}) and (\mathrm{LogsSpec}) pin output/log canonicalization.
A capsule is self‑contained relative to determinism: everything that can affect the outcome must be pinned or embedded; otherwise it cannot certify OK.
A canonical schema requires (unless policy explicitly permits omission):
[
{\mathrm{TargetAddr},\ \mathrm{RouterProfileDigest},\ \mathrm{CorridorProfileDigest},\ \mathrm{EnvPins},\ \mathrm{Seeds},\ \mathrm{Inputs},\ \mathrm{Steps},\ \mathrm{ExpectedDigests},\ \mathrm{Outputs},\ \mathrm{Logs}}.
]
12.1.2 ReplayRecipe (\mathsf{RRc}) and strict ordering
Define the replay program carried inside (\mathsf{RC}) as:
[
\mathsf{RRc}:=(\mathrm{StepList},\mathrm{StepPins},\mathrm{StepDigests},\mathrm{OrderingPolicy})
]
where:
(\mathrm{StepList}=(s_1,\dots,s_n)) is a sequence of named steps (parse, canonicalize, compute NF, apply kernel, decode stream, compute digest, compare),
(\mathrm{StepPins}) lists required profile digests per step,
(\mathrm{StepDigests}) optionally pins intermediate digests for localization,
(\mathrm{OrderingPolicy}) is pinned; steps execute in strict order.
The recipe forbids best‑effort skipping; if a step cannot run, outcome is AMBIG/FAIL according to the corridor policy.
12.1.3 EnvironmentPin (\mathsf{EPn}) and numeric determinism
An EnvironmentPin is the minimal set of version identifiers needed to make replay deterministic across machines:
[
\mathsf{EPn}:=(\mathrm{OSID},\mathrm{ArchID},\mathrm{RuntimeID},\mathrm{LibDigests},\mathrm{Compiler/VMID?},\mathrm{FloatingPolicy},\mathrm{EndianPolicy},\mathrm{SchemaPins}).
]
Missing environment pins prevent OK certification; at best the result is AMBIG (recoverable) or FAIL (policy forbids).
12.1.4 DeterminismChecklist (\mathsf{DCk}): what must be pinned
A DeterminismChecklist is an explicit list of “knobs that must be pinned” for a capsule class:
[
\mathsf{DCk}:=(\mathrm{CapsuleClass},\mathrm{RequiredPins},\mathrm{OptionalPins},\mathrm{ForbiddenUnpinned},\mathrm{FallbackPolicy}),
]
including forbidden unpinned dependencies such as unpinned RNG, time dependence, network dependence, or unspecified platform behavior. (\mathsf{DCk}) is itself pinned by digest and referenced by verifiers.
12.1.5 Verifier engine (V): deterministic replay execution and failure classification
Fix a pinned verifier engine version (\mathrm{VerifierVer}). The verifier executes (\mathsf{RC}) as a deterministic state machine whose internal state includes step index, loaded pins, loaded inputs, digest accumulator, transcript buffer, and budget counters (\chi).
Given (\mathsf{RC}), (V) must:
validate pins via the registry snapshot (existence, digest match, not revoked),
validate schema versions (capsule schema, container schema),
load inputs (embedded or fetched by digest under pinned policy),
execute steps in order producing intermediate digests,
compare observed digests to (\mathrm{ExpectedDigests}),
emit a canonical transcript and a truth classification under corridor mapping.
Mismatch handling is strict:
FAIL if mismatch is deterministic under a pinned environment,
CONFLICT if mismatches disagree across independently pinned replays (fork indicator),
AMBIG only if evaluation cannot be completed due to missing pins/inputs and recovery is permitted.
Mismatch cannot be ignored or downgraded to OK.
