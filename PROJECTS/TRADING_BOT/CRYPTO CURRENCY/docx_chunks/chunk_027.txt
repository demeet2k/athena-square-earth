□ H.S — Square: Coupling Contracts, Product-Structure Invariants, Multi-Rail Constraints
H.S1 — Objects
H.S1.a — Axis index set and state carriers
Fix dimension (d\ge 1). Let ([d]:={1,\dots,d}). Axis-coupled structures are defined over:
Phase torus (\mathbb{T}^d) or its discretizations (\Sigma_{B_d}).
Square carrier (\mathcal{S}{d,k}) and its factorization into ((\mathbb{Z}{4^k})^d) or ((\mathbb{Z}_4)^{dk}) under pinned decomposition.
Triangle carrier ((\mathbb{Z}_3)^d) (Option A) or (\mathbb{Z}_3) (Option B) under pinned mode policy.
H.S1.b — Rail system and multi-rail constraints
A rail is a pinned control schedule defining an axis-wise sequencing regime. A rail system is a tuple:
[
\mathsf{Rails}:=\langle \mathsf{RailID},\ \mathsf{LaneSet},\ \mathsf{ScheduleSpec},\ \mathsf{PinsDigest}\rangle
]
where:
LaneSet is a finite set of lane identifiers (e.g., Su/Me/Sa) or extended lane alphabets pinned by policy.
ScheduleSpec is a deterministic mapping:
[
\mathsf{Sched} : (\mathsf{PulseID}, k, \mathsf{StateSummary}) \to \mathsf{LaneSel}
]
with (\mathsf{LaneSel}) a finite selection describing which lane applies on which axis and which coupling constraints activate.
A multi-rail constraint is a predicate restricting simultaneous lane assignments across axes. Multi-rail constraints are part of coupling contracts and must be pinned.
H.S1.c — CouplingContract
A CouplingContract is the meaning-bearing object defining when axes may be treated as independent and when they are coupled.
[
\mathsf{CouplingContract}:=
\langle
\mathsf{CoupleVer},;
\mathsf{Dim},;
\mathsf{CoupleMode},;
\mathsf{AxisPartition},;
\mathsf{AllowedCouplings},;
\mathsf{DisallowedCouplings},;
\mathsf{WitnessRequirements},;
\mathsf{TieBreakSpec},;
\mathsf{PinsDigest},;
\mathsf{SchemaPins},;
\mathsf{Sig}
\rangle
]
Where:
CoupleMode ∈ {PRODUCT, BLOCK, GLOBAL}.
AxisPartition partitions ([d]) into disjoint blocks (B_1,\dots,B_m), pinned.
PRODUCT: all blocks are singletons.
BLOCK: blocks can be >1.
GLOBAL: a single block ([d]).
AllowedCouplings and DisallowedCouplings are finite sets of coupling patterns expressed as typed constraints over operations (tunnel steps, lifts, projections, and scheduling).
WitnessRequirements is a pinned list of required witness families to assert coupling legality (see H.S2 and H.F).
H.S1.d — Multi-axis tunnel step object
A multi-axis tunnel step is a vector:
[
\delta \in \mathbb{T}^d
]
or a discrete ring step on (\Sigma_{B_d}) with associated lift/projection references. The canonical step object is:
[
\mathsf{TunnelStep} :=
\langle
\mathsf{StepVer},;
\mathsf{Dim},;
\mathsf{StepVector},;
\mathsf{AxisSupport},;
\mathsf{StepSetDigest},;
\mathsf{RailSelDigest},;
\mathsf{PinsDigest},;
\mathsf{StepDigest}
\rangle
]
AxisSupport is the set of axes where (\delta_i\neq 0).
StepSetDigest pins the allowed step set (\Delta\mathcal{H}^{(d)}) (AppG/Ch06).
RailSelDigest pins the lane selection applicable at the step.
StepDigest is content-addressed.

H.S2 — Operators
H.S2.a — Coupling legality predicate
Coupling legality is a deterministic predicate:
[
\mathsf{LegalCouple} :
(\mathsf{CouplingContract},\mathsf{StateSummary},\mathsf{TunnelStep},\mathsf{CorridorID}) \to {\mathsf{true},\mathsf{false}}.
]
A tunnel step is legal under coupling if and only if all the following hold:
AxisSupport respects AxisPartition: coupling constraints must allow simultaneous nonzero steps on the axis set selected.
Multi-rail constraints: lane assignments on the axis set satisfy pinned multi-rail constraints.
Allowed step membership: (\delta\in\Delta\mathcal{H}^{(d)}) (or equivalent discrete step membership) under pinned StepSetDigest.
Corridor compliance: the step stays inside the pinned corridor neighborhood on (\mathbb{T}^d) or ring representation.
Required topology receipts (spin/holonomy) are satisfied when noncommuting schedules could yield path dependence (H.F2).
If any required condition cannot be decided due to missing pins, the result is AMBIG (by emitting evidence plan under AppL) rather than assuming legality.
H.S2.b — Product-structure factorization operator
For CoupleMode=PRODUCT, the operator family must factorize across axes:
For any axis-separated operators (A_i) acting on axis (i), the combined operator satisfies:
[
A = \bigotimes_{i=1}^d A_i
]
or, for generators in Kronecker sum form:
[
G = \bigoplus_{i=1}^d G_i.
]
Coupling contracts must declare which factorization law applies. Any deviation from declared factorization is a coupling violation unless explicitly migrated.
H.S2.c — Block-coupling operator
For CoupleMode=BLOCK, axes in a block (B_j) may be treated as coupled. The coupled operator is constrained to act only within the block’s subspace and must be representable as:
tensor product across blocks, and
arbitrary (but pinned) operator inside each block.
Formally:
[
\mathsf{Op} = \bigotimes_{j=1}^m \mathsf{Op}{B_j}
]
with each (\mathsf{Op}{B_j}) pinned by digest.
H.S2.d — Cross-axis inference prohibition operator
Define the prohibition rule:
[
\mathsf{Infer}(i \Rightarrow j) \text{ is forbidden unless }\mathsf{CoupleMode}\neq \mathsf{PRODUCT}\text{ and explicit witness exists.}
]
This is enforced by requiring that any inference edge whose source and target addresses lie on different axis blocks must reference a CouplingWitness proving that coupling is active and legal for the inference.

H.S3.b — No illegal cross-axis inference
Any claim that transfers information from axis (i) to axis (j) across different coupling blocks without a CouplingWitness is invalid for OK routing.
H.S3.c — Multi-axis tunnel legality invariant
A multi-axis tunnel step is admissible only if:

H.S3.d — Deterministic tie-break invariant
Any ambiguity in axis ordering, lane ordering, or step selection must be resolved by a pinned tie-break. Unpinned tie-breaks produce AMBIG.

□ I.S — Square: CorridorProfile, Budget Discipline, and Routability Semantics
I.S1 — Objects
I.S1.a — Truth lattice and routability threshold
Truth typing is a closed vocabulary:
[
\mathsf{TruthType} := {\mathsf{OK},\mathsf{NEAR},\mathsf{AMBIG},\mathsf{FAIL},\mathsf{CONFLICT}}.
]
For routability and aggregation, corridors define a strict preorder (pinned):
[
\mathsf{OK} \succ \mathsf{NEAR} \succ \mathsf{AMBIG} \succ \mathsf{FAIL} \succ \mathsf{CONFLICT}.
]
A corridor defines a minimum admissible truth threshold:
[
\mathsf{MinTruth}(\mathsf{Corridor}) \in {\mathsf{OK},\mathsf{NEAR},\mathsf{AMBIG}}.
]
Paths whose composite truth is below (\mathsf{MinTruth}) are not routable under that corridor.
I.S1.b — Budget vector
A corridor budget is:
[
\mathsf{Budget} := \langle \kappa,\beta,\delta,\chi \rangle
]
with semantics:
(\kappa): commutation residual budget (holonomy/spin allowance),
(\beta): drift/approximation residual budget,
(\delta): recursion depth budget,
(\chi): compute/search/replay budget.
Budgets are interpreted only under pinned budget semantics. A corridor must pin the aggregation mode for each component:
additive, max, or mixed (e.g., max for (\delta), additive for (\chi)).
I.S1.c — CorridorProfile schema (mandatory)
A CorridorProfile is the meaning-bearing object that defines legality, truth typing, budgets, and escalation.
[
\mathsf{CorridorProfile} :=
\langle
\mathsf{CorrVer},;
\mathsf{CorridorID},;
\mathsf{Scope},;
\mathsf{PinsDigest},;
\mathsf{BudgetCaps},;
\mathsf{BudgetAggSpec},;
\mathsf{AllowedEdgeKinds},;
\mathsf{RequiredArtifacts},;
\mathsf{TypingRules},;
\mathsf{EscalationRules},;
\mathsf{PromotionRules},;
\mathsf{DisclosureRules},;
\mathsf{QuarantineRules},;
\mathsf{RoutabilityRules},;
\mathsf{SchemaPins},;
\mathsf{Sig}
\rangle
]
Required fields:
CorridorID: content-addressed digest of CorridorProfile with CorridorID field zeroed.
Scope: {LOCAL, INTER} plus scope bindings (manuscript IDs if INTER).
PinsDigest: pin closure required to interpret this corridor (AppA/AppD).
BudgetCaps: (\mathsf{Budget}) caps.
BudgetAggSpec: per-component aggregation mode and charging rules.
AllowedEdgeKinds: mapping from truth class to allowed edge kinds.
RequiredArtifacts: mapping from truth class to mandatory artifact families.
TypingRules: deterministic rule set mapping evidence to truth typing.
EscalationRules: deterministic mapping specifying when failures trigger fences/quarantines.
PromotionRules: deterministic obligations for NEAR→OK and AMBIG→NEAR/OK transitions.
DisclosureRules: mandatory disclosure formats and propagation rules.
QuarantineRules: deterministic quarantine triggers and propagation rules.
RoutabilityRules: path acceptance predicates, tie-breaks, and thresholds.
SchemaPins: schema closure required to interpret all corridor artifacts.
I.S1.d — Allowed edge kinds per truth class
CorridorProfile must define a total function:
[
\mathsf{AllowedEdgeKinds} : \mathsf{TruthType} \to \mathcal{P}(\mathsf{EdgeKind})
]
Minimum requirement:
(\mathsf{AllowedEdgeKinds}(\mathsf{OK})) must exclude any edge kind that requires missing proofs or missing pins.
(\mathsf{AllowedEdgeKinds}(\mathsf{NEAR})) must require residual ledger edge kinds and disclosure edges where appropriate.
(\mathsf{AllowedEdgeKinds}(\mathsf{AMBIG})) must require EvidencePlan edges.
(\mathsf{AllowedEdgeKinds}(\mathsf{FAIL})) must require FailureWitness or ConflictPacket edges if reproducible.
(\mathsf{AllowedEdgeKinds}(\mathsf{CONFLICT})) must require fence/quarantine propagation edges.
I.S1.e — Required artifact families per truth class
CorridorProfile must define:
[
\mathsf{RequiredArtifacts} : \mathsf{TruthType} \to \mathcal{P}(\mathsf{ArtifactKind})
]
Minimum required artifacts:
OK: ReplayCapsule, WitnessBundle, PinClosure, DigestCommitments.
NEAR: all OK artifacts plus ResidualLedger and Expiry/renewal obligations.
AMBIG: EvidencePlan, bounded candidate set, discriminators, routes to evidence.
FAIL: FailureWitness and/or ReproRecipe (unless policy forbids non-reproducible failures).
CONFLICT: ConflictPacket, FenceSpec, QuarantineRecord.
