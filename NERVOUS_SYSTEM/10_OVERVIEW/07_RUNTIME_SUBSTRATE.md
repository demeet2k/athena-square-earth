# RUNTIME SUBSTRATE

## Purpose

The nervous system is not only a set of canonical documents. It also needs a live substrate where swarm packets, waves, sessions, and receipts can be written without polluting the canonical cortex.

## Three-Surface Contract

### Cortex

Location:

- `NERVOUS_SYSTEM/`

Function:

- publish stable maps
- hold schemas and manifests
- expose the canonical reading order

### Runtime Hub

Location:

- `self_actualize/mycelium_brain/`
- `self_actualize/mycelium_brain/nervous_system/`

Function:

- hold active swarm files
- carry live receipts
- preserve the zero-point toolkit and prompt contracts
- expose runtime folders for ganglia, neurons, packets, swarms, waves, ledgers, and sessions

### Governance Mirror

Location:

- `ECOSYSTEM/NERVOUS_SYSTEM/`

Function:

- preserve reusable framework definitions
- expose bridge readmes for agents that enter through the ecosystem surface
- keep older conceptual work addressable without forking the canonical cortex

## Runtime Folder Set

The current runtime substrate uses these folders under `self_actualize/mycelium_brain/nervous_system/`:

- `ganglia/`
- `neurons/`
- `packets/`
- `receipts/`
- `sessions/`
- `swarms/`
- `waves/`
- `ledgers/`

## Writeback Law

Every serious run should produce:

1. one runtime artifact in the runtime hub
2. one canonical map, schema, ledger, or manifest update in the cortex
3. one bridge or governance note if routing law changed

If a run only updates one surface, the nervous system drifts.

## Example Flow

```
runtime seed -> pod packet -> wave packet -> contraction -> cortex manifest -> governance mirror note
```

## Current Operational Meaning

The runtime substrate is now the lawful place to store higher-dimensional swarm artifacts. The cortex records what the runtime has stabilized. The governance mirror keeps the routing language reusable for sibling agents.
